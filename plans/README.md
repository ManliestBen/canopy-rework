## Findings considered — backlog (grounded, not yet planned)

These are real, vetted findings that didn't make the top-6 cut. Promote to a
plan when you want them.

- **SEC-05 — weather/photos store raw upstream error text** served to clients
  (`services/weather.ts:174`, `services/photos.ts:78-81`). Lower risk than
  the ICS case (operator-configured URLs, behind auth). Plan 002 sanitizes
  the ICS/Google path; extend the same treatment here. Effort S.
- **CORRECTNESS-04 — one bad stored setting resets ALL settings to defaults**
  (`services/settings.ts:27-28`: whole-object `safeParse` → `DEFAULT_SETTINGS`
  on any failure). Parse per-key with per-key fallback + a log. Effort S–M,
  MED confidence (writes are validated, so it only bites on schema drift).
- **CORRECTNESS-03 — Slideshow crossfade inner `setTimeout` never cleared**
  (`features/screensaver/Slideshow.tsx:31-42`): orphaned timeout fires after
  unmount / dep change on the 24/7 kiosk. Effort S.
- **CORRECTNESS-06 — non-transitive event sort comparator**
  (`services/eventCache.ts:236-240`): all-day-first rule only applies within
  equal `startKey`, violating the comparator contract → engine-dependent
  ordering in edge cases. Cosmetic. Effort S.
- **CORRECTNESS-07 — single-day chore chart doesn't advance at midnight**
  (`features/household/ChoresPage.tsx:19-21`): `dateKey` initialized once;
  kiosk left overnight shows yesterday until tapped. LOW confidence (may be
  intended nav semantics — confirm desired behavior first). Effort S.
- **DEPS-01 — replace 116 MB `googleapis` metapackage** with scoped
  `@googleapis/calendar` + `@googleapis/gmail` + `google-auth-library`

- **DEPS-02 — precompile the server for production** instead of running
  `tsx src/index.ts` (`server/package.json` `start`/`build`); `build` is
  `tsc --noEmit` so there's no runnable artifact. Effort M, MED confidence on
  magnitude (wants an on-Pi RSS/boot measurement).

## Direction findings (maintainer's call — options, not defects)

Ordered by grounding strength. Each cites repo evidence; effort is coarse.

- **DIR-01 — grow backup coverage + add scheduled backups.** Export today is
  only settings + users (`routes/backup.ts:19-24`), yet the schema comment
  (`shared/src/schemas/backup.ts:6`) says "future sections appended as
  optional keys." Tasks, chores, star balances/redemptions, lists, meals,
  calendar sources, announcements all live in SQLite but are excluded — a
  family that "restored their backup" after an SD failure loses all of it.
  Plan 006 fixes the *data-loss bug* in restore but deliberately does NOT
  expand coverage (that needs per-domain merge-vs-replace + FK-ordering
  decisions). Highest-value direction item; the data model already exists.
  Coarse effort M–L. → design/spike plan when ready.
- **DIR-02 — inbound email-to-add.** Gmail is send-only
  (`services/gmail.ts`: `sendEmail`/`gmailConfigured`, no receive); PLAN.md
  Phase 7 stretch names "inbound email-to-add." The OAuth identity is already
  wired, lowering marginal cost — "forward a school email → it becomes an
  event/to-do." Needs a safe parse→review pipeline for untrusted mail. Coarse
  effort L. → spike.
- **DIR-03 — pluggable photo sources (local/NAS).** `services/photos.ts` has
  a single Cloudinary provider; PLAN.md notes "photo sources pluggable
  (local/NAS on roadmap)." A `PhotoSource` interface (list/stream) is one
  small refactor away and removes a cloud dependency for a device already
  sitting on the LAN next to a NAS. Coarse effort M, LOW risk (additive,
  Cloudinary stays default). → design.
- **DIR-06 — downloadable full-database backup/restore (owner request, stretch).**
  Requested 2026-07-15: a Settings action to **download the current SQLite
  database as a file** and restore the app from it later. Distinct from the
  existing JSON config backup (settings + members only) and from plan 006
  (which fixed restore atomicity): this captures *everything* — chores, star
  balances, lists, meals, calendar sources, announcements — as a single
  restorable artifact. Deferred to the stretch list until the current work is
  merged. Design notes for whoever picks it up: stream a consistent snapshot
  (better-sqlite3 `db.backup()` into a temp file, or a WAL-checkpoint + file
  copy) rather than copying the live file mid-write; on restore, validate it's
  a Canopy DB (check `user_version` / a known table), apply it atomically, and
  decide the running-process reload story (simplest: write the file and
  require a restart). Keep the PIN-hash-in-the-file privacy note in mind — the
  download is sensitive, unlike the secret-free JSON config backup. Coarse
  effort M.
- **DIR-05 — `deviceName` is a near-no-op setting** (`schemas/settings.ts`
  defines it; only `SettingsPage.tsx:109-110` reads it, to edit itself —
  never shown in header/logs/session labels). Either wire it (header +
  `createSession` label — effort S) or pursue the larger "multi-device
  naming" story from PLAN Phase 8. A persisted-but-inert setting erodes trust.

## Not reproduced / rejected (so nobody re-audits)

- **"Expensive recurrence expansion per client render"** — NOT present.
  Recurrence is expanded server-side (Google `singleEvents:true`; ICS via
  node-ical `.between()`) and cached already-expanded. Client `recurrence.ts`
  only classifies rules for the edit modal. No finding.
- **Loopback trust spoofable via `X-Forwarded-For`** — NO. `isLoopback`
  reads `req.socket.remoteAddress` with `app.set('trust proxy', false)`
  (`app.ts:29`), so headers can't forge the peer. By-design and safe.
- **Session fixation / weak PIN hashing / SQL injection / XSS** — checked,
  all clean: 32-byte tokens stored as SHA-256, fresh per login, scrypt +
  `timingSafeEqual` on the PIN, parameterized SQL throughout (incl. the
  dynamic `IN (...)` list), no `dangerouslySetInnerHTML`. No findings.
- **`happy-dom` critical npm-audit advisories** — dev-only test dependency,
  not in the production runtime path. Routine bump, not an incident (folded
  into the DEPS backlog, not a security plan).
- **better-sqlite3 sync access / HA deferral / Cloudinary-over-Photos /
  tsx-run server as an architecture choice** — documented PLAN.md tradeoffs,
  not findings. (The *production* `tsx` runtime cost is a separate, narrower
  perf item — DEPS-02 above.)

## Environment note for executors

`npm test` at the repo root may fail in the server workspace with
`NODE_MODULE_VERSION ... better_sqlite3.node` if your shell's Node differs
from the one that built the native module. Fix once with
`npm rebuild better-sqlite3` (CI uses Node 22; local Node 20 or 22 is fine).
This is not a code defect — client and shared tests pass regardless.
